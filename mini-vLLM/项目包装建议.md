## 三、改 vLLM 源码的具体方案

### 3.1 workload-aware Scheduler

#### 要解决的问题

Fab 场景下不同任务有不同需求：

| 任务 | 特点 | 需求 |
|------|------|------|
| 告警分诊 | 短 prompt + 短 output | 低延迟（TTFT） |
| 交接班摘要 | 中 prompt + 中 output | 均衡 |
| 报告生成 | 长 prompt + 长 output | 高吞吐 |
| 多轮诊断 | 多轮对话，共享 prefix | prefix cache 命中率 |

#### 改动文件

**文件 1：`vllm/v1/engine/__init__.py`**
EngineCoreRequest 加 `workload_type` 字段：

```python
@dataclass
class EngineCoreRequest:

    prompt_token_ids: list[int]
    sampling_params: SamplingParams
    arrival_time: float
    priority: int = 0
    workload_type: str = "default"   # ← 新增
```

**文件 2：`vllm/v1/request.py`**
Request 加 `workload_type` 字段：

```python

        ...

```

**文件 3：`vllm/v1/core/sched/scheduler.py`**
`schedule()` 加分层调度逻辑：

```python
class Scheduler:
    def schedule(self) -> SchedulerOutput:
        # 分两阶段：latency-sensitive 先调度，throughput-oriented 后调度
        latency_sensitive = [r for r in self.waiting
                           if r.workload_type in ("alarm_triage", "diagnosis")]
        throughput_first = [r for r in self.waiting
                          if r.workload_type in ("report_generation", "batch_summary")]

        # 方案 A：Priority 分层（简单，~20 行改动）
        # latency-sensitive → priority=0（高）
        # throughput-first  → priority=1（低）
        # PriorityQueue 自动排序，高优先先调度

        # 方案 B：Token Budget 分配（中等，~80 行改动）
        # 每步 token budget 按比例分：60% 给 latency，40% 给 throughput
        # latency 请求先调度，用完 60% 后 throughput 再调度

        # 方案 C：动态切换（高级，~120 行改动）
        # 监控当前队列状态：如果 waiting 队列长 → 切 throughput 模式
        # 如果 waiting 队列短 → 切 latency 模式
```

**文件 4：`vllm/v1/core/sched/request_queue.py`**（可选）
加 `LatencyQueue` / `ThroughputQueue`：

```python
class WorkloadAwareQueue:
    def __init__(self):
        self.latency_queue = PriorityQueue()    # 告警分诊、诊断
        self.throughput_queue = PriorityQueue()  # 报告生成、批量总结

    def add(self, request: Request):
        if request.workload_type in ("alarm_triage", "diagnosis"):
            self.latency_queue.add(request)
        else:
            self.throughput_queue.add(request)
```

### 3.2 Prefix Cache 优化

#### 要解决的问题

多轮诊断场景：

```
Turn 1: "光刻区机台报错 stage position drift，分析可能原因"
Turn 2: "补充信息：该机台 24 小时内第 3 次报错"    ← 共享 Turn 1 的 prefix
Turn 3: "建议排查步骤？"                           ← 共享 Turn 1+2 的 prefix
```

vLLM 现有的 prefix cache 按 **block hash** 匹配。如果 prompt 格式略有不同（比如多了个时间戳），hash 就对不上。

#### 方案 A：Session-aware prefix cache（推荐，改动适中）

**文件：`vllm/v1/core/kv_cache_manager.py`**

```python
class KVCacheManager:
    def __init__(self):
        ...
        self.session_blocks: dict[str, list[int]] = {}  # ← 新增

    def free(self, request_id: str, session_id: str = None):
        if session_id:
            # 多轮对话：不释放 block，保留给下一轮用
            self.session_blocks[session_id] = self.req_to_blocks[request_id]
        else:
            # 单轮：正常释放
            blocks = self.req_to_blocks.pop(request_id)
            self.free_blocks.extend(blocks)

    def get_computed_blocks(self, request) -> int:
        # 先查 session cache
        if request.session_id and request.session_id in self.session_blocks:
            # 用 session 里保留的 block 做 prefix 匹配
            ...
        # 再查全局 prefix cache
        ...
```
