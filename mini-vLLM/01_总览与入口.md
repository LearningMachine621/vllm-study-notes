# 01 总览与入口：LLM → LLMEngine 调用链

> 对标 vLLM：`AsyncLLM` → `EngineCore` → `EngineCoreProc`

## 一、调用链全景

```
用户代码
  │
  ▼
LLM(llm.py:4)                    ← 唯一对外接口，继承 LLMEngine
  │
  ▼
LLMEngine(llm_engine.py:15)       ← 引擎主循环
  ├── __init__()
  │     ├── Config(model_path)     ← 解析 HF config
  │     ├── spawn TP workers       ← torch.multiprocessing.spawn
  │     ├── ModelRunner(rank=0)    ← GPU 执行器
  │     ├── AutoTokenizer          ← HF tokenizer
  │     └── Scheduler              ← 调度器
  │
  ├── add_request(prompt, params)
  │     ├── tokenizer.encode()
  │     └── scheduler.add(Sequence)
  │
  ├── step()                       ← 核心一步
  │     ├── scheduler.schedule()   → (seqs, is_prefill)
  │     ├── model_runner.run()     → token_ids
  │     └── scheduler.postprocess()
  │
  └── generate(prompts, params)    ← 批量生成
        ├── add_request() × N
        └── while not finished: step()
```

## 二、关键设计决策

### 2.1 LLM 是空壳

```python
# llm.py —— 5 行
class LLM(LLMEngine):
    pass
```

为什么？因为 vLLM 的 `AsyncLLM` 也是一层薄封装。nano-vllm 保持同样的 API 风格：
用户只看到 `LLM`，内部全是 `LLMEngine`。

### 2.2 多进程 Tensor Parallel

```python
# llm_engine.py:24-31
ctx = mp.get_context("spawn")
for i in range(1, config.tensor_parallel_size):
    event = ctx.Event()
    process = ctx.Process(target=ModelRunner, args=(config, i, event))
    process.start()
    self.ps.append(process)
    self.events.append(event)
self.model_runner = ModelRunner(config, 0, self.events)
```

关键点：
- **rank 0 在主进程**，rank 1~N 是子进程
- 用 `spawn` 不用 `fork`（CUDA context 不能 fork）
- `Event` 用于同步：rank 0 写共享内存 → set event → rank 1~N 读
- 主进程直接调 `ModelRunner`，子进程走 `loop()` 等待指令

### 2.3 step() 是最小调度单元

```python
def step(self):
    seqs, is_prefill = self.scheduler.schedule()     # 1. 调度
    token_ids = self.model_runner.call("run", seqs, is_prefill)  # 2. 执行
    self.scheduler.postprocess(seqs, token_ids, is_prefill)      # 3. 后处理
    outputs = [(seq.seq_id, seq.completion_token_ids) for seq in seqs if seq.is_finished]
    return outputs, num_tokens
```

每一步要么是 **prefill**（处理 prompt tokens），要么是 **decode**（生成一个 token）。
与 vLLM 的 `EngineCore.step()` 完全对应。

## 三、与 vLLM 对照

| nano-vllm | vLLM v1 | 差异 |
|-----------|---------|------|
| `LLM` | `AsyncLLM` | nano-vllm 无 async/await |
| `LLMEngine` | `EngineCore` + `EngineCoreProc` | 合二为一，无 IPC 层 |
| `step()` | `EngineCore.step()` | 逻辑相同，vLLM 多了 output_processor |
| `mp.Process` | `mp.Process` + ZMQ | nano-vllm 用 SharedMemory + Event |
| `generate()` | `EngineCore.generate()` | nano-vllm 同步阻塞，vLLM 异步流式 |

## 四、代码量对比

| 文件 | nano-vllm | vLLM 对应 | vLLM 行数 |
|------|----------|----------|----------|
| llm.py | 5 行 | engine/async_llm.py | ~500 |
| llm_engine.py | 90 行 | engine/core.py | ~2100 |
| **总计** | **95 行** | | **~2600 行** |

nano-vllm 用 95 行实现了 vLLM 2600 行的核心功能。
省掉的是：async、ZMQ IPC、DP/PP、multiprocess_mode 工厂方法。

## 五、学习要点

1. **入口极简**：`LLM(LLMEngine)` 就是全部接口
2. **step() 是灵魂**：schedule → execute → postprocess 三步循环
3. **多进程是可选的**：TP=1 时主进程自己跑，TP>1 时 spawn 子进程
4. **generate() 是批处理**：把 N 个 prompt 塞进去，while 循环 step() 直到全部完成
