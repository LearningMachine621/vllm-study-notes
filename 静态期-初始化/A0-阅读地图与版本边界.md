# A0 · 阅读地图、版本边界与启动里程碑

> **版本基准**：vLLM `0.24.0`，V1 Engine。
>
> **主路径**：`vllm serve <model>`、单 API Server、单 DP rank、`AsyncMPClient`、CUDA GPU、标准生成模型。
>
> **阅读规则**：代码位置以“文件 + 类/函数名”为准，行号只作 v0.24.0 快照参考；vLLM 变化很快，不把行号当稳定接口。

---

## 1. 本知识库回答什么

这里的“静态初始化”指：

> 从命令行参数进入 Python，到 HTTP 服务真正开始监听之间，只在启动期完成一次、并在后续多个推理 step 中复用的对象创建、资源发现、模型加载、显存规划、通信建链和服务装配。

它包括：

- CLI 参数解析与 `VllmConfig` 构造；
- API Server 进程中的 `AsyncLLM` 前端组件；
- `EngineCoreClient` 的 ZMQ 通道与 EngineCore 子进程；
- Executor、Worker、GPUModelRunner 和模型权重；
- KV Cache profiling、容量计算、物理 tensor 分配、compile/warmup；
- Scheduler、KVCacheManager、BlockPool 等逻辑资源管理器；
- FastAPI、OpenAI Serving 对象及 uvicorn 监听。

它不包括：

- 单个请求的 tokenize、入队、schedule、forward、sample、detokenize；
- 每个 step 临时创建的 token budget、scheduled request 集合等；
- 请求到来后才触发的真正按需初始化，例如某些 structured-output 后端；
- shutdown、sleep/wake、在线扩缩容的完整运行时协议。

运行期只在需要解释“某个静态对象为什么存在”时作为预览出现。

## 2. 三条不能混淆的关系线

### 2.1 启动时间线

```text
CLI
  → AsyncEngineArgs
  → VllmConfig
  → AsyncLLM 前端轻组件
  → AsyncMPClient / ZMQ / spawn
  → EngineCoreProc
  → EngineCore
  → Executor / Worker / GPUModelRunner / model
  → KV Cache profile + allocate + warmup
  → Scheduler / BlockPool
  → READY handshake
  → FastAPI / Serving
  → uvicorn listen
```

### 2.2 OOP 继承

```text
EngineClient
└── AsyncLLM

EngineCoreClient
├── InprocClient
└── MPClient
    ├── SyncMPClient
    └── AsyncMPClient
        └── DPAsyncMPClient
            └── DPLBAsyncMPClient

EngineCore
└── EngineCoreProc
```

关键点：

- `AsyncLLM` **不是** `EngineCoreClient` 的子类。
- `AsyncLLM.engine_core` 持有一个 `EngineCoreClient` 实现；在线默认通常是 `AsyncMPClient`。
- `EngineCoreProc` 才是子进程中的 ZMQ 外壳，它继承并复用 `EngineCore` 的业务逻辑。

### 2.3 运行时持有关系

```text
FastAPI app.state / OpenAIServing*
└── AsyncLLM
    ├── renderer
    ├── input_processor
    ├── output_processor
    ├── logger_manager
    └── engine_core: AsyncMPClient
        ├── ZMQ sockets
        ├── serializers
        └── EngineCore 子进程句柄
            └── EngineCoreProc
                └── EngineCore
                    ├── model_executor
                    │   └── Worker(s)
                    │       └── GPUModelRunner + model
                    ├── Scheduler
                    │   └── KVCacheManager
                    │       └── BlockPool
                    └── 辅助结构
```

继承回答“它是什么”，持有关系回答“谁创建谁、谁控制谁”。排查启动问题时，后者通常更重要。

## 3. 进程与线程地图

### API Server 进程

- CLI / FastAPI / uvicorn；
- `AsyncLLM`；
- `AsyncMPClient`；
- ZMQ `ROUTER`、`PULL` socket；
- 异步输出拉取 task；
- EngineCore 进程管理和存活监控。

### EngineCore 子进程

- `EngineCoreProc` 主线程；
- `EngineCore`；
- 对应执行后端的 Executor；
- 单进程后端时可能直接持有本地 Worker；
- input socket thread：ZMQ → `input_queue`；
- output socket thread：`output_queue` → ZMQ；
- 主线程中的 busy loop。

### Worker 进程

- `UniProcExecutor`：Worker 与 EngineCore 同进程；
- `MultiprocExecutor`：按并行拓扑再创建 Worker 进程；
- Ray / external launcher：生命周期由相应后端管理。

所以“vLLM 是两进程”只适用于最简 `AsyncMPClient + UniProcExecutor` 视图；TP/PP/Ray 下是多级进程拓扑。

## 4. 全局 milestone

milestone 必须表示一个可验证的完成条件，而不是“某函数刚被调用”。

| 标志 | 完成条件 | 主要产物 |
|---|---|---|
| 🚩 M0 | 版本与主路径边界确定 | 本文的分析前提 |
| 🚩 M1 | CLI 参数已转换为 `AsyncEngineArgs` | 轻量输入交接边界 |
| 🚩 M2 | `create_engine_config()` 返回 | 经验证、派生后的 `VllmConfig` |
| 🚩 M3 | AsyncLLM 的三组前端基础完成 | 配置/跨进程准备、运行证据策略、renderer/processors |
| 🚩 M4 | 前端 ZMQ endpoint 绑定且 EngineCore 进程已启动 | IPC 地址、socket、进程句柄 |
| 🚩 M5 | EngineCoreProc 启动握手和进程级环境就绪 | 地址、identity、DP 进程环境 |
| 🚩 M6 | Worker/设备/分布式通信环境就绪 | CUDA device、进程组、GPUModelRunner |
| 🚩 M7 | Executor 构造返回且模型加载完成 | GPU/设备上的模型权重 |
| 🚩 M8 | KV specs、可用显存与逻辑容量确定 | `KVCacheConfig`、`num_gpu_blocks` |
| 🚩 M9 | KV 物理存储注册且模型 compile/warmup 完成 | KV tensors、已预热执行路径 |
| 🚩 M10 | Scheduler 和逻辑 KV 管理链完成 | queues、KVCacheManager、BlockPool |
| 🚩 M11 | EngineCoreProc I/O 线程就绪，READY 已发出；随后进入 busy loop | 可服务的 proc 端 |
| 🚩 M12 | client 收齐 READY，AsyncLLM 构造完成 | 可用的 `AsyncLLM` |
| 🚩 M13 | FastAPI state 和 OpenAI Serving 对象完成 | HTTP 业务层 |
| 🚩 M14 | uvicorn 开始监听 | 对外可连接的服务 |

注意：

- M8 与 M9 刻意拆开：知道“能分多少块”不等于 GPU 上的 KV tensor 已分配且执行路径已 warmup。
- M10 是逻辑账本就绪；物理显存属于 M9。
- M4 到 M12 期间前端通常阻塞在启动握手，HTTP 尚未监听。

## 6. 事实层级

本文用三种标签避免把主干简化误当成普遍真理：

- **主干**：上述 v0.24.0 在线服务路径直接发生；
- **分支**：仅在 DP/TP/PP、Ray、P/D、多模态、投机解码等配置下发生；
- **运行期预览**：启动时只创建对象，请求到来后才执行其核心逻辑。

源代码校验优先级：

1. v0.24.0 源码中的符号和调用关系；
2. v0.24.0 API 文档；
3. 官方架构文档；
4. 本知识库中的解释性伪代码。

官方基准入口：

- [vLLM v0.24.0 源码标签](https://github.com/vllm-project/vllm/tree/v0.24.0)
- [v0.24.0 API 文档](https://docs.vllm.ai/en/v0.24.0/api/)
- [V1 Process Architecture](https://docs.vllm.ai/en/latest/design/arch_overview/)

知识库中的代码块若标注“示意”，只表达顺序与所有权，不保证逐字符等同源码。
