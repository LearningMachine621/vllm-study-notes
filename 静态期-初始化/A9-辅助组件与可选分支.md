# A9 · 辅助组件与可选初始化分支

> **阶段**：主要发生在 M2–M10 各节点  
> **目的**：把主干时间线与按配置启用的分支分开，避免把可选对象写成必然对象。

---

## 1. StructuredOutputManager

EngineCore 在 KV 初始化后、Scheduler 前创建管理器：

```text
StructuredOutputManager(vllm_config)
```

静态期通常完成：

- 保存配置；
- 选择/记录 grammar backend 策略；
- 准备 grammar bitmask 管理；
- 准备异步 grammar compilation 能力；
- 建立 request grammar 状态入口。

真正 backend、grammar/FSM 的构建可能在第一个结构化输出请求到来后才发生。

准确区分：

```text
manager 已初始化                    backend/grammar 按需初始化
M10 之前                            运行期
```

不能因为 backend 延迟创建，就说 StructuredOutputManager 不属于静态初始化。

---

## 2. multimodal receiver cache

多模态输入可能在前端完成预处理或通过 tensor IPC 传输特征。EngineCore 的 receiver cache：

- 对齐前端发送与 proc 接收的 feature；
- 避免重复传输/处理；
- 为 Request 构造提供本地 feature；
- 与 Worker/ModelRunner 的 encoder cache 分工。

主干纯文本模型中它可能为空或轻量 no-op。

---

## 3. request_block_hasher

当 prefix caching 或 connector 需要稳定 block identity 时创建。

要求：

- token block 内容相同且前缀链相同 → hash 一致；
- 不同 model/adapter/multimodal 上下文不能错误复用；
- 支持特殊 `None`/起始 hash；
- hash block size 与 KV block/group 语义兼容。

它负责“这个请求的逻辑块是谁”，BlockPool 负责“这个 hash 当前对应哪个物理 slot”。

---

## 4. abort queue

EngineCore 的 abort queue 允许 abort 与 model execution 解耦：

- I/O/control 路径投递 request IDs；
- EngineCore 在安全点批量处理；
- Scheduler 释放逻辑 blocks；
- connector 执行 request-finished hook；
- 输出正确 finish reason。

queue 是静态对象，单次 abort 消息是运行期数据。

---

## 5. batch queue

`max_concurrent_batches > 1` 时创建：

- 支持 schedule 与 execute 流水；
- PP 下减少 pipeline bubble；
- 使 `step_fn` 选择 `step_with_batch_queue`；
- 保存 future、SchedulerOutput、execute future 的组合。

它不是普通请求 waiting queue，而是“已 schedule batch 的执行流水队列”。

---

## 6. Observability

### 前端

- tracer；
- API/AsyncLLM spans；
- StatLoggerManager；
- 前端 request metrics。

### EngineCore/Worker

- Engine iteration stats；
- scheduler/KV/cache metrics；
- model execution/compile time；
- 可选 worker tracer。

配置统一来自 `observability_config`，但对象分散在多个进程；不能说“tracer 只在 AsyncLLM 初始化一次”。

---

## 7. Compilation/CUDA Graph

配置在 M2 建立，相关对象可能在 M6 的 GPUModelRunner 创建，真正 compile/warmup/capture 主要在 M9。

```text
CompilationConfig                  M2
dispatcher/buffer 准备             M6
compile/warmup/capture             M9
graph replay                       运行期
```

将这四层分开，才能准确解释 `enforce_eager`、compile level 和 capture sizes 的影响。

---

## 8. 多模态分支

跨阶段变化：

| 阶段 | 新增/变化 |
|---|---|
| M2 | model/multimodal limits/config |
| M3 | renderer、InputProcessor 多模态解析 |
| M4–M5 | tensor IPC queue/receiver |
| M6 | GPUModelRunner encoder/input/cache |
| M8 | profiling 包含 encoder peak |
| M10 | encoder cache manager/budget |

多模态不是只在 InputProcessor 增加一个字段，而是贯穿前端、IPC、Worker、profiling、Scheduler。

---

## 9. 投机解码分支

| 阶段 | 新增/变化 |
|---|---|
| M2 | `SpeculativeConfig`，验证 target/draft 兼容 |
| M6 | drafter、spec buffers、sampling/verification 结构 |
| M8/M9 | profiling/warmup 覆盖额外执行路径 |
| M10 | Scheduler/EngineCore 检查 draft tokens |
| 运行期 | propose → target verify → accept/reject |

它不改变 M4 的 ZMQ 基本两通道，但会改变 EngineCoreOutputs 和 step 内部语义。

---

## 10. DP/TP/PP 分支

### DP

- 多个 EngineCoreProc；
- 每个 rank 独立 Scheduler/KV Cache/模型执行资源；
- client/coordinator 路由；
- READY 必须按受管 ranks 汇聚。

### TP

- 单个模型 shard 分布到多个 Worker；
- M6 建 TP group；
- M7 每 rank 加载 shard；
- forward 中 collective；
- KV specs/capacity 按 rank 汇聚。

### PP

- 不同 Worker 持有不同层；
- M6 建 PP group；
- M7 加载 stage shard；
- batch queue/async scheduling 可能启用；
- KV groups 按 stage 对应。

---

## 11. Ray 与 external launcher

| 维度 | 本地 MP | Ray | external launcher |
|---|---|---|---|
| Worker 创建者 | Executor/进程管理器 | Ray | torchrun/外部系统 |
| 控制 RPC | 本地 MP | Ray RPC/actor | 外部 world + executor |
| rank 来源 | 本地配置 | placement/actor | 环境/world |
| failure | process monitor | Ray actor state | launcher/collective |

主干知识应复用“完成条件”，不要硬套本地 `multiprocessing.Process` 的每一步。

---

## 12. P/D 分离与 KV Connector

新增三组关系：

```text
Scheduler connector                 逻辑调度/load-save 状态
Worker connector                    物理 KV 发送/接收
EngineCore metadata aggregation     连接 PP/TP ranks 的握手信息
```

启动时：

- M2 构造 transfer config；
- M6 Worker 创建 connector 侧资源；
- M9 KV tensors 注册后才能建立物理映射；
- M10 Scheduler connector 创建；
- EngineCore 汇聚 Worker handshake metadata 并写入 Scheduler connector。

因此 connector 的“初始化完成”跨越多个基础 milestone。

---

## 13. LoRA

启动期主要：

- M2 创建/校验 `LoRAConfig`；
- M3 InputProcessor 支持 LoRA request；
- M6 ModelRunner 准备 manager/slots；
- Scheduler/EngineCore 保存相关能力。

具体 adapter 常在启动参数或运行期动态 load。不要把“LoRA 能力初始化”与“所有 adapter 权重已加载”混为一谈。

---

## 14. Prefix caching

跨阶段：

- M2：开关/hash algo；
- M8：验证模型因果性，必要时关闭；
- M10：request hasher、BlockPool cache map；
- 运行期：hash、match、refcount、cache/evict。

---

## 15. feature 完成条件矩阵

| Feature | 静态初始化何时可认为完成 |
|---|---|
| structured output | manager 和 Scheduler 接口就绪；grammar backend 可延迟 |
| multimodal | 前端/IPC/ModelRunner/encoder budget 全部就绪 |
| spec decode | drafter/target 路径已在 M9 warmup，EngineCore 标志就绪 |
| DP | 所有受管 Engine 达到 M11，client 完成 M12 |
| TP/PP | 所有 Worker 达到 M7/M9，collective/batch queue 就绪 |
| P/D | worker + scheduler connectors 及 metadata handshake 完成 |
| LoRA | manager/slots 能力就绪；具体 adapter 另算 |

---

## 16. Milestone 归属

可选 feature 不新增一条平行的 M15+ 时间线，而是挂到基础节点：

- 配置归 M2；
- 前端适配归 M3；
- IPC 资源归 M4/M5；
- Worker 对象归 M6/M7；
- 内存和 warmup 归 M8/M9；
- 调度/逻辑 manager 归 M10；
- 跨进程 ready 归 M11/M12。

这样即使 feature 组合变化，仍能用统一 milestone 判断启动进度。
