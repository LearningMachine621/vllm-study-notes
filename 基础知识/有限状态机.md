# 有限状态机（Finite State Machine, FSM）

## 一、什么是有限状态机

有限状态机是一种计算模型，用来描述一个系统在**有限个状态**之间如何**根据输入事件**进行**转移**。

三个核心要素：

```
1. 状态（State）    — 系统在某一时刻所处的情况
2. 事件（Event）    — 触发状态变化的输入或条件
3. 转移（Transition）— 在某个事件发生时，从一个状态切换到另一个状态
```

用数学语言说，FSM 是一个五元组：

$$M = (S, \Sigma, \delta, s_0, F)$$

- $S$：有限状态集合
- $\Sigma$：有限输入字母表（事件集合）
- $\delta: S \times \Sigma \rightarrow S$：转移函数
- $s_0 \in S$：初始状态
- $F \subseteq S$：终止状态集合

---

## 二、最简单的例子：红绿灯

```
状态集合:  {绿灯, 黄灯, 红灯}
事件集合:  {定时器到}
转移规则:
  绿灯 → 黄灯  (定时器到)
  黄灯 → 红灯  (定时器到)
  红灯 → 绿灯  (定时器到)
```

用图表示：

```
    ┌──────────┐
    ↓          │
 ┌──────┐  定时器  ┌──────┐  定时器  ┌──────┐
 │ 绿灯 │ ──────→ │ 黄灯 │ ──────→ │ 红灯 │
 └──────┘        └──────┘        └──────┘
                                  │
                      定时器 ←────┘
```

关键特征：
- 任何时刻只能处于**一个状态**
- 状态转移是**瞬时的**（不考虑过渡时间）
- 转移只取决于**当前状态 + 输入事件**，不依赖历史（无记忆性，称为"马尔可夫性"）

---

## 三、FSM 的两种类型

### 3.1 接受器 / 识别器（Acceptor）

判断输入序列是否"合法"。有 accept / reject 两种输出。

```
例子：判断字符串是否以 "ab" 结尾

状态: {S0, S1, S2, Accept}
输入: {a, b, 其他}

S0 --a--> S1
S0 --其他--> S0
S1 --b--> S2 (Accept)
S1 --a--> S1
S1 --其他--> S0
S2 --a--> S1
S2 --其他--> S0
```

### 3.2 转换器（Transducer）

在状态转移的同时产生输出。vLLM 的 scheduler 就是这种——每一轮调度不仅改变请求状态，还产生调度决策（哪些请求被 admit、哪些被 preempt）。

## 四、vLLM Scheduler 的状态机

### 4.1 请求状态定义

vLLM V1 的请求状态定义在 `vllm/v1/request.py`：

```python
class RequestStatus(enum.IntEnum):
    WAITING = auto()                            # 等待调度
    WAITING_FOR_STRUCTURED_OUTPUT_GRAMMAR = auto()  # 等待 grammar（结构化输出）
    WAITING_FOR_REMOTE_KVS = auto()             # 等待远程 KV 传输
    WAITING_FOR_STREAMING_REQ = auto()          # 等待流式输入
    RUNNING = auto()                            # 正在执行（GPU 上计算）
    PREEMPTED = auto()                          # 被抢占（KV 已释放）
    # --- 以下为终态 ---
    FINISHED_STOPPED = auto()                   # 正常结束（遇到 EOS）
    FINISHED_LENGTH_CAPPED = auto()             # 达到最大长度
    FINISHED_ABORTED = auto()                   # 被外部中止
    FINISHED_IGNORED = auto()                   # 被忽略
    FINISHED_ERROR = auto()                     # 出错
    FINISHED_REPETITION = auto()                # 重复检测终止
```

**判断是否结束的函数：**
```python
def is_finished(self) -> bool:
    return self.status > RequestStatus.PREEMPTED
```

### 4.2 状态转移图

```
                         ┌─────────────────────────────────────────┐
                         │              WAITING 区域                │
                         │                                         │
                         │  ┌──────────┐                           │
              ┌──────────┤  │ WAITING_ │                           │
              │          │  │ FOR_     │ ← grammar/stream/KV 就绪  │
              │          │  │ BLOCKED  │ ──────────────┐           │
              │          │  └──────────┘               │           │
              │          │       ↑                     ↓           │
              │          │  ┌──────────┐         ┌──────────┐      │
              │          │  │ WAITING  │ ←────── │ PREEMPTED│      │
              │          │  └──────────┘         └──────────┘      │
              │          │       │                     ↑           │
              │          └───────│─────────────────────│───────────┘
              │                  │ allocate_slots 成功  │ KV 分配失败
              │                  ↓                     │
              │          ┌──────────┐                  │
              │          │ RUNNING  │ ─────────────────┘
              │          └──────────┘
              │                  │
              │                  │ 遇到 EOS / 达到 max_len / abort / error
              │                  ↓
              │          ┌──────────────────┐
              └────────→ │   FINISHED_*     │
                         └──────────────────┘
```

### 4.3 关键转移详解

#### 转移 1: WAITING → RUNNING（调度）

**触发条件：** `schedule()` 方法中，`kv_cache_manager.allocate_slots()` 成功

**发生的事情：**
1. Scheduler 从 `self.waiting` 队列取出请求
2. 计算需要的 token 数：`num_new_tokens = num_tokens_with_spec - num_computed_tokens`
3. 检查 token budget：不能超过 `max_num_batched_tokens`
4. 调用 `kv_cache_manager.allocate_slots()` 申请 KV blocks
5. 申请成功 → 请求进入 `self.running`，状态设为 `RUNNING`
6. 请求被加入 `scheduled_new_reqs`（新请求）或 `scheduled_resumed_reqs`（被抢占后恢复的请求）

**代码位置：** `vllm/v1/core/sched/scheduler.py` line 826

#### 转移 2: RUNNING → PREEMPTED（抢占）

**触发条件：** 另一个请求需要 KV blocks，但当前显存不够

**发生的事情：**
1. Scheduler 选择优先级最低的 running 请求
2. 调用 `_preempt_request()`：
   - 释放该请求的所有 KV blocks（`kv_cache_manager.free()`）
   - 释放 encoder cache（如果有）
   - 状态设为 `PREEMPTED`
   - `num_computed_tokens` 重置为 0（之前算的 KV 全丢了）
   - `num_preemptions` 计数器 +1
3. 请求被**前插**到 `self.waiting` 队列头部（优先恢复）

**代码位置：** `vllm/v1/core/sched/scheduler.py` line 952-972

**为什么是 RECOMPUTE 而不是 SWAP？**
vLLM V1 默认使用 RECOMPUTE 策略：被抢占的请求丢弃所有 KV，下次重新计算。这比 SWAP（把 KV 换出到 CPU 内存）更简单，因为在 V1 架构下重新计算的开销更低。

#### 转移 3: RUNNING → FINISHED_*（结束）

**触发条件：** 生成了 EOS token / 达到 max_tokens / 被外部 abort / 出错

**发生的事情：**
1. 请求从 `self.running` 移除
2. 释放所有 KV blocks
3. 状态设为对应的 FINISHED 子状态
4. 结果通过 API 返回给客户端

**代码位置：** `vllm/v1/core/sched/scheduler.py` line 1750

#### 转移 4: WAITING → WAITING_FOR_BLOCKED（阻塞等待）

**触发条件：** 请求需要结构化输出 grammar / 远程 KV 传输 / 流式输入

**发生的事情：**
1. 请求从 `self.waiting` 移到 `self.skipped_waiting`
2. 每轮调度时，Scheduler 会检查阻塞条件是否解除
3. 条件解除后，请求回到 `self.waiting` 或 `PREEMPTED`（取决于之前的状态）

**代码位置：** `vllm/v1/core/sched/scheduler.py` line 2061

### 4.4 调度的两阶段

`schedule()` 方法分两个阶段：

```
Phase 1: 处理 RUNNING 请求
  ├── 遍历 self.running
  ├── 计算每个请求需要的新 token 数
  ├── 尝试 allocate_slots()
  ├── 失败 → 抢占最低优先级的 running 请求 → 重试
  └── 成功 → 保持 RUNNING 状态

Phase 2: 处理 WAITING / PREEMPTED 请求（仅在没有抢占发生时执行）
  ├── 遍历 self.skipped_waiting（阻塞请求，尝试解除阻塞）
  ├── 遍历 self.waiting
  ├── 计算 cached tokens（prefix cache 命中）
  ├── 尝试 allocate_slots()
  ├── 失败 → break（本轮不再 admit 新请求）
  └── 成功 → WAITING/PREEMPTED → RUNNING
```

**为什么要"先 RUNNING 后 WAITING"？**
- 保证已经在跑的请求优先获得资源（减少延迟抖动）
- 新请求只有在有剩余资源时才能被 admit
- 这就是 decode-first 策略

---

### 4.5 队列结构

```python
self.waiting: deque[Request]           # 正常等待队列
self.skipped_waiting: deque[Request]   # 阻塞等待队列
self.running: deque[Request]           # 正在执行队列
```

**调度优先级（FCFS 模式）：**
```
skipped_waiting 的头部 > waiting 的头部
即：被阻塞的请求优先解除阻塞
```

**调度优先级（PRIORITY 模式）：**
```
比较两个队列头部的优先级，取更高的那个
```

## 六、FSM 与 vLLM 调度器的关系总结

| FSM 概念 | vLLM 对应 |
|---------|----------|
| 状态 | RequestStatus（WAITING/RUNNING/PREEMPTED/FINISHED） |
| 事件 | allocate_slots 成功/失败、EOS、abort、grammar 就绪 |
| 转移函数 | `schedule()` 方法 |
| 初始状态 | WAITING（请求刚到） |
| 终止状态 | FINISHED_* |
| 状态机实例 | 每个请求都是一个 FSM 实例 |
| 调度器角色 | 集中式的状态机控制器，每轮决定所有请求的状态转移 |

**核心理解：** Scheduler 不是在"调度请求"，而是在"管理一组并发的有限状态机"。每轮 `schedule()` 调用，就是对所有请求做一轮状态转移。资源充足就推进（WAITING→RUNNING），资源不足就回退（RUNNING→PREEMPTED），完成就终止（→FINISHED）。

## 七、面试角度

**Q: vLLM 的请求有哪些状态？**
A: V1 下主要有 WAITING、RUNNING、PREEMPTED 和多个 FINISHED 子状态。还有几个阻塞等待状态（grammar、远程 KV、流式输入）。

**Q: 什么是 Preemption？vLLM 用什么策略？**
A: 当显存不够时，Scheduler 会抢占优先级最低的 running 请求，释放其 KV blocks 给新请求用。V1 默认用 RECOMPUTE（被抢占的请求下次重新计算 KV），不是 SWAP（换出到 CPU）。因为 V1 架构下重算开销更低。

**Q: Scheduler 每一轮做什么？**
A: 两阶段。Phase 1 处理 RUNNING 请求（保证 decode 继续），如果 KV 不够就抢占；Phase 2 处理 WAITING/PREEMPTED 请求（admit 新请求），按 token budget 和 KV blocks 限制决定放多少进来。

**Q: 为什么要 decode-first？**
A: 已经在跑的请求（decode）对延迟更敏感——用户正在等下一个 token。新请求（prefill）还没开始，等一轮影响不大。所以优先保证 decode 的 KV 分配。

## 八、与你项目的衔接

你后续做 scheduler decision trace 时，核心就是记录每一轮 `schedule()` 中的状态转移：

```python
# 在 schedule() 中添加 trace
trace = {
    "round": self._schedule_round,
    "running_count": len(self.running),
    "waiting_count": len(self.waiting),
    "token_budget_remaining": token_budget,
    "kv_cache_usage": kv_cache_usage,
    "admitted": [],      # WAITING → RUNNING 的请求
    "preempted": [],     # RUNNING → PREEMPTED 的请求
    "skipped_reason": {}, # 未被 admit 的原因
}
```

这就是你 fork patch 的核心产出。
