# 进程与对象 — vLLM 多进程架构解析

> 核心问题：为什么 vLLM 要把 API Server 和 EngineCore 放在不同进程里？
> `EngineCoreProc` 的意义是什么？

## 1. 基本概念

### 对象（Object）

- 内存里的一坨数据 + 方法
- 同一进程里的对象可以直接互相调用（函数调用）
- 通过变量名引用对方

### 进程（Process）

- 操作系统分配的**独立内存空间**
- 不同进程里的对象**互相看不到对方**
- 必须通过 IPC（进程间通信）传递数据

```
同进程：
┌──────────────────────────────────┐
│         一个 Python 进程          │
│                                  │
│  对象A ──直接调用──► 对象B        │
│  共享同一块内存                   │
└──────────────────────────────────┘

多进程：
┌──────────────────┐    IPC     ┌──────────────────┐
│  Python 进程 A    │  ◄──────► │  Python 进程 B    │
│  各自独立的内存    │   网络通信  │  各自独立的内存    │
└──────────────────┘            └──────────────────┘
```

## 2. vLLM 里的两种模式

### 模式 A：InprocClient（同进程）

```
┌──────────────────────────────────────────┐
│              pid=10744（一个进程）          │
│                                          │
│  API Server (AsyncLLM)                   │
│    └── self.engine_core                  │
│         = InprocClient                   │
│           └── self.engine_core           │
│                = EngineCore  ◄── 真正的引擎 │
│                                          │
│  调用方式：client.engine_core.step()      │
│  = 直接执行 EngineCore.step()（函数调用）   │
└──────────────────────────────────────────┘
```

- API Server 和 EngineCore 在**同一个进程**里
- 直接函数调用，没有通信开销
- **问题**：GPU forward 会阻塞 API 收请求

### 模式 B：MPClient（多进程）

```
┌──────────────────┐   ZMQ    ┌──────────────────┐
│  pid=10744       │ ◄──────► │  pid=10746       │
│  父进程          │  网络通信  │  子进程           │
│                  │          │                  │
│  API Server      │          │  EngineCoreProc  │
│  + MPClient      │          │  (继承EngineCore) │
│  (只有socket)     │          │  (持有GPU资源)    │
└──────────────────┘          └──────────────────┘
  没有EngineCore实例！            真正的引擎在这里
```

- API Server 在父进程，EngineCore 在**子进程**
- 通过 ZMQ（vLLM）或 pipe（Python）通信
- **好处**：GPU forward 不阻塞 API 收请求

## 3. EngineCoreProc 是什么

`EngineCoreProc` = `EngineCore` + ZMQ 通信层

```python
# vllm/v1/engine/core.py L806
class EngineCoreProc(EngineCore):  # 继承 EngineCore
    def __init__(self, ...):
        super().__init__(...)      # 初始化 EngineCore（模型、KV Cache、Scheduler）
        # 额外：设置 ZMQ socket、握手

    def run_busy_loop(self):       # 额外：忙循环，等消息 → 处理 → 返回
        while True:
            msg = self.input_socket.recv()
            # ... 处理请求 ...
            self.output_socket.send(result)
```

**它只在子进程里出现。** InprocClient 模式不需要它，因为不需要 IPC。

## 4. 为什么只能在两个地方创建 EngineCore

vLLM 全球只有**两处**实例化 EngineCore：

| 位置 | 文件 | 模式 |
|------|------|------|
| `InprocClient.__init__` | `core_client.py L285` | 同进程，直接创建 |
| `EngineCoreProc.run_engine_core` | `core.py L1110` | 子进程入口，子进程里创建 |

**为什么不能在别处创建？**

| 约束 | 原因 |
|------|------|
| 不能在 API Server 里直接 new 后传给子进程 | GPU 显存不能 pickle，不能跨进程传递 |
| 不能在父进程创建后 fork 给子进程 | CUDA context 不能 fork，必须在子进程里自己初始化 |
| 不能拷贝多份 | GPU 资源独占，一个 EngineCore 管一块 GPU |

所以设计成：**谁用 EngineCore，谁就在自己的进程里创建它。**

## 5. 为什么要多进程

### 同进程的问题

```
时间线（同进程）：
  t=0   请求A到达 → API Server 调 engine.step()
  t=200ms  GPU forward 还没结束...
  t=300ms  请求B到达 → API Server 卡在 step() 里，无法处理！
  t=500ms  step() 返回 → 才能处理请求B
```

### 多进程的解决

```
时间线（多进程）：
  t=0   请求A到达 → MPClient 通过 ZMQ 发给子进程 → 立刻返回
  t=1ms 请求B到达 → MPClient 继续收请求 → 通过 ZMQ 发给子进程
  t=200ms 子进程处理完请求A → 返回结果给父进程
  t=201ms 子进程处理请求B...
```

父进程（API Server）永远不被 GPU 计算阻塞。

## 6. 进程 vs 线程

| | 进程 | 线程 |
|--|------|------|
| 内存 | 独立内存空间 | 共享内存 |
| 通信 | IPC（pipe/ZMQ/socket） | 直接读写共享变量 |
| GIL | 各自独立的 GIL | 受同一个 GIL 限制 |
| 创建开销 | 大（要分配内存） | 小 |
| vLLM 用法 | API Server ↔ EngineCore | EngineCore 内部 IO 线程 |

Python 的 GIL（全局解释器锁）意味着同一进程里的多线程**不能并行执行 Python 代码**。所以 vLLM 用多进程来实现真正的并行：父进程处理 HTTP，子进程跑 GPU 计算。

## 7. 最小代码演示

```python
import multiprocessing
import os

class EngineCore:
    def __init__(self, name):
        self.name = name
        self.pid = os.getpid()
    def step(self):
        return f"{self.name} stepped at pid={self.pid}"

# ── 模式 A：同进程 ──
class InprocClient:
    def __init__(self):
        self.engine_core = EngineCore("Inproc")  # 同进程创建
    def step(self):
        return self.engine_core.step()  # 直接调用

# ── 模式 B：多进程 ──
def subprocess_entry(pipe):
    engine = EngineCore("MP")  # 子进程里创建
    pipe.send({"ready": True})
    while True:
        msg = pipe.recv()
        if msg["cmd"] == "step":
            pipe.send({"result": engine.step()})
        elif msg["cmd"] == "quit":
            break

class MPClient:
    def __init__(self):
        parent, child = multiprocessing.Pipe()
        self.pipe = parent
        self.proc = multiprocessing.Process(target=subprocess_entry, args=(child,))
        self.proc.start()
        self.pipe.recv()  # 等 READY

    def step(self):
        self.pipe.send({"cmd": "step"})
        return self.pipe.recv()["result"]

# 运行
client_a = InprocClient()
print(f"同进程: {client_a.step()}")  # pid 相同

client_b = MPClient()
print(f"多进程: {client_b.step()}")  # pid 不同
```

输出：
```
同进程: Inproc stepped at pid=10744    ← 主进程 pid
多进程: MP stepped at pid=10746        ← 子进程 pid，不同！
```

## 8. Python multiprocessing 常用方法

### 8.1 Process — 创建子进程

```python
from multiprocessing import Process

def worker(name):
    print(f"子进程 {name}, pid={os.getpid()}")

p = Process(target=worker, args=("worker-1",))
p.start()      # 启动子进程
p.join()       # 等待子进程结束
p.pid          # 子进程的 pid
p.is_alive()   # 是否还在运行
p.terminate()  # 强制终止
```

### 8.2 start_method：spawn / fork / forkserver

`multiprocessing.set_start_method()` 决定**子进程怎么创建**。

| 方法 | 机制 | 平台 | 特点 |
|------|------|------|------|
| `fork` | 复制父进程内存（copy-on-write） | Linux 默认 | 快，但可能继承父进程的脏状态 |
| `forkserver` | 先 fork 一个干净的服务进程，再从它 fork | Linux | 比 fork 干净 |
| `spawn` | 全新启动一个 Python 解释器，只执行 target 函数 | **macOS 默认**、Windows | 最干净，但最慢 |

```python
import multiprocessing
multiprocessing.set_start_method("spawn")  # 必须在最开始调用，只调一次
```

**为什么 macOS 必须用 spawn？**

macOS 从 High Sierra 开始，默认 start method 改为 `spawn`。原因是 `fork` 在 macOS 上不安全：
- Objective-C 运行时不支持 fork 后继续使用
- CUDA context 不能 fork（GPU 驱动限制）
- fork 后子进程可能继承父进程的锁，导致死锁

**spawn 的工作流程：**

```
父进程                          子进程
  │                               │
  │ Process(target=fn, args=...)  │
  │─── spawn ───────────────────► │ 全新 Python 解释器
  │                               │ import 必要模块
  │                               │ 执行 target(*args)
  │                               │
  │                               │ 不继承父进程内存！
  │                               │ 只有 target 函数需要的东西
```

所以 `spawn` 模式下：
- 子进程不会继承父进程的全局变量
- `target` 函数和它的参数必须可 pickle
- 类的实例方法不能直接作为 target（模块级函数才行）

**vLLM 为什么用 spawn？**

```python
# vllm/v1/engine/utils.py CoreEngineProcManager
context = multiprocessing.get_context("spawn")
context.Process(target=EngineCoreProc.run_engine_core, ...)
```

因为 CUDA context 不能 fork。必须在子进程里重新初始化 GPU。

### 8.3 Pipe — 双向通信

```python
from multiprocessing import Pipe

parent_conn, child_conn = Pipe()  # 双向管道
parent_conn.send({"cmd": "step"})
msg = child_conn.recv()           # {"cmd": "step"}
child_conn.send({"result": 42})
result = parent_conn.recv()       # 42
```

### 8.4 Queue — 多生产者多消费者

```python
from multiprocessing import Queue

q = Queue()
q.put("hello")
item = q.get()  # "hello"
```

Queue 是线程/进程安全的，适合多对多通信。vLLM 用 `torch.multiprocessing.Queue` 传递 tensor。

### 8.5 shared_memory — 共享内存（Python 3.8+）

```python
from multiprocessing import shared_memory

shm = shared_memory.SharedMemory(create=True, size=1024)
shm.buf[0] = 42  # 直接写入共享内存
# 其他进程可以通过 name 访问同一个 shm
shm.close()
shm.unlink()
```

vLLM 用共享内存传递大 tensor（避免序列化开销）。

### 8.6 Lock / Event / Semaphore — 同步原语

```python
from multiprocessing import Lock, Event, Semaphore

lock = Lock()
with lock:  # 互斥锁
    # 临界区

event = Event()
event.set()      # 通知
event.wait()     # 等待通知

sem = Semaphore(3)  # 最多 3 个并发
sem.acquire()
sem.release()
```

### 8.7 对照 vLLM 的 IPC 选择

| Python 原语 | vLLM 用法 | 原因 |
|------------|----------|------|
| `Pipe` | 本例演示用 | 简单，但只有两个端点 |
| `Queue` | tensor 传递（`torch.multiprocessing.Queue`） | 多生产者多消费者 |
| `shared_memory` | multimodal tensor IPC | 零拷贝，避免序列化 |
| `ZMQ` (第三方) | **主要通信方式** | 高性能、支持多种模式（REQ/REP、PUB/SUB、ROUTER/DEALER） |
| `msgpack` (第三方) | 序列化 | 比 pickle 快 20x |

## 9. 总结

```
你的迷你实现 vllmserving_v1：
  main.py → EngineCore(...)  ← InprocClient 模式，同进程

vLLM 生产环境：
  AsyncLLM → EngineCoreClient(AsyncMPClient)
           → spawn 子进程
           → 子进程里创建 EngineCoreProc(EngineCore)
           → ZMQ 通信

为什么？
  GPU 计算耗时长，不能阻塞 API 收请求 → 必须多进程
  CUDA context 不能 fork → 必须 spawn
  跨进程不能直接调用 → 必须 IPC（ZMQ）
```
